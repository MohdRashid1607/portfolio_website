Glowing was built with one guiding principle: commercial realism over experimental novelty. Rather than building a novel or abstract project, the aim was to produce something that reflects the standards of real professional skincare e-commerce — a platform that genuinely looks and feels like it belongs on the commercial web.

The design is deliberately minimal — a restrained palette of neutral tones with soft green and gold accents. Green for natural ingredients and wellness; gold used sparingly for quality and luxury. The Urbanist typeface was selected for its modern readability across all screen sizes.

"Rather than pursuing experimental novelty, this project intentionally prioritised commercial realism — prioritising reliability and maintainability over conceptual novelty, which reflects professional development environments."

— Critical Reflection, M.Rashid

Three conceptual principles guided every decision: simplicity (clean interface, clear navigation, no clutter), trust (security-first architecture, professional design), and quality (premium visual execution that feels credible from the first interaction).

The intended audience is modern skincare consumers who shop online and expect a smooth, high-quality experience — busy professionals who value fast purchasing, and users who value product information and brand credibility. Every feature was built to serve that user.

// Features

What Glowing Does

Auth — JWT + Google OAuth

Email/password authentication with JWT for stateless session management, plus Google OAuth via Passport.js. Passwords hashed with bcrypt before database storage.

PayPal Sandbox Payments

Integrated PayPal JavaScript SDK in sandbox mode — full order creation and transaction confirmation flow without processing live financial data.

Cloudinary Image Hosting

Product images uploaded to and served from Cloudinary's CDN — efficient global content delivery and no server storage overhead.

Filter · Sort · Paginate

Full product browsing system with filtering by category, sorting by price/rating, and pagination — replicating real e-commerce product discovery.

Persistent Cart & Wishlist

Shopping cart and wishlist stored in localStorage — persisting across pages and sessions using custom JavaScript classes for state synchronisation.

Admin Dashboard — RBAC

Role-based access control protects admin routes. Administrators can manage products, orders, and users through a dedicated panel — separated from user-facing functionality.

User Dashboard

Account details, order history, and profile management — the complete user account experience expected by modern e-commerce shoppers.

Fully Responsive

Optimised for desktop, tablet, and mobile. Mobile-friendly navigation with collapsible menus, dynamic layouts, and thumb-friendly interaction targets.

Blog · Newsletter · Trust

Blog section for skincare content, newsletter subscription, brand value sections (100% Organic · Cruelty-Free · Trusted Ingredients) — the full marketing layer.

// Design Process

Wireframes & Design Iteration

Homepage · Shop Page Wireframes
Homepage (left) — hero slider, product grid, category sections. Shop page (right) — filter sidebar, product grid, pagination.
Login · About Page Wireframes
Login/Signup page (left) — contact info, form, map. About page (right) — brand story, team section, milestone timeline.

Wireframes were intentionally low-fidelity — focused on layout structure, not colours or typography. Their purpose was to establish visual hierarchy and identify potential usability issues before writing a single line of code.

One significant design change during development: the homepage hero section was originally a static layout. During implementation it was redesigned as a slider to allow multiple featured products to be displayed in the same space — improving marketing flexibility while maintaining a clean layout. Product cards were also iterated: initially all interactions were always visible, which felt crowded. Revised so secondary actions (wishlist, add-to-cart) only appear on hover — reducing visual noise significantly.

// Engineering

Key Technical Decisions

1

Vanilla JavaScript over React / Vue

Deliberately chose plain JavaScript to strengthen foundational understanding — modular design, state management, and event-driven behaviour without abstraction layers. Implementing cart and wishlist as custom classes reinforced understanding of data persistence and UI synchronisation at a level frameworks would have hidden. The trade-off: longer development time for cross-page state management, which would favour a framework in a larger project.

2

JWT Authentication — Stateless by Design

JWT provides stateless session management aligned with the RESTful architecture. A signed token is issued on login and included in subsequent request headers. This simplifies horizontal scaling but introduces known limitations — no built-in token revocation or refresh mechanisms — which would require resolution before production deployment.

3

MVC over Microservices

MVC architecture was chosen over microservices or serverless functions due to the scope of a single-contributor project. It provides a clear, organised codebase with effective separation of concerns, without the operational complexity of distributed services. Under high traffic, service decomposition would become necessary — but for a proof of concept, MVC offered the right balance of clarity and flexibility.

4

PayPal Sandbox — Demonstrable Without Risk

PayPal's sandbox mode demonstrates a complete end-to-end payment flow — order creation, confirmation, success handling — without processing live financial data. The decision acknowledges this is conceptual rather than production-ready, flagging PCI-DSS compliance and fraud prevention as required additions for a live deployment.

5

Cloudinary for Media — CDN at No Cost

Rather than storing product images on the server, Cloudinary enables global CDN delivery and eliminates server storage overhead — a production-standard approach even in an academic context. The trade-off is vendor dependency, which introduces considerations around service availability and long-term cost in a commercial scenario.

// Reflection

Key Learnings

01

Building a real product forces real engineering decisions

Choosing between JWT and sessions, MVC and microservices, Vanilla JS and React — these weren't academic exercises. They were trade-offs with real consequences for performance, scalability, and maintainability. Making them consciously, and being able to justify them, is what professional development looks like.

02

Vanilla JavaScript teaches you what frameworks hide

Implementing cart state synchronisation across pages without React revealed exactly what component-based frameworks solve — and why they exist. Understanding the problem first, then reaching for the abstraction, produces far better developers than starting with the tool.

03

Design and development must be planned together

The wireframing phase directly shaped the data models and API structure. Pages with filter + pagination required specific MongoDB query design from the start — not as an afterthought. Separating design from development planning is the mistake that causes rework.

04

Security isn't an add-on — it's an architecture decision

bcrypt, JWT, Google OAuth, RBAC, environment variables — every security layer had to be designed into the architecture from day one. Retrofitting security onto an existing system is exponentially harder than building with it in mind from the start.

05

The gap between "proof of concept" and "production" is clearly defined

This project deliberately identified what it would take to make Glowing a real commercial product: token refresh, PCI-DSS compliance, automated testing, CI/CD pipelines, rate limiting, monitoring. Knowing the gap is the first step to closing it — and that self-awareness is the most valuable output of the project.
