// Overview

About This Project

Code Lab 2 was the moment Python stopped being an exercise language and became a real creative tool. The brief was to build a data-driven application — and the result was a fully interactive Pokémon desktop app with four distinct modules, a themed GUI, and background music.

The application was built in 8 days following a structured development plan — from initial API research and wireframing through to testing, debugging, and a final user walkthrough video. Every module required a different set of Python skills: from data retrieval and randomisation to GUI event handling and music playback.

"The creation of the Pokémon Data-Driven Application was a thought-provoking journey that combined creativity with technical problem solving — proving that Python can be as expressive as it is functional."

— Critical Reflection, M.Rashid

// App Modules

4 Core Features

📖 Pokédex

Search and retrieve Pokémon information from a preloaded dataset. Users can look up stats, types, and details for any Pokémon — functioning as a full in-app encyclopaedia. The foundation module that all others build upon.

❓ Quiz Game

A Pokémon-themed multiple choice quiz with randomised questions pulled from the dataset. Tracks the player's score across all questions and displays the final result at the end — fixed after a label update bug was debugged.

⚔️ Battle Simulator

Simulates a one-off battle between two Pokémon using randomised stats and battle logic. Calculates the winner and displays the result — giving users an interactive way to pit Pokémon against each other.

🏆 Battle League

A multi-round competitive mode where Pokémon battle through a full league bracket. Tracks progression round-by-round and displays results after each match — rebuilt after the "Next Round" button bug was identified and fixed.

// Debugging Log

Bugs Fixed

Background Music Silent on Launch

✓ Fixed

Background music failed to play on application launch despite being correctly loaded in the project directory. No console errors were shown — the failure was completely silent.

Fixed by reinitialising the pygame.mixer module in the setup code before playback begins — ensuring music only starts if it isn't already active.

Quiz Final Score Not Displaying

✓ Fixed

After the final quiz question, the score was not shown — the label remained blank and the user had no way to see their result, making the quiz feel broken at the end.

A missing widget update after the last question was identified. Added a score calculation and label update after every question — correctly displaying the final result on completion.

Battle League "Next Round" Unresponsive

✓ Fixed

The "Next Round" button in the Battle League became unresponsive after the first click — trapping users in round one and making the entire league mode non-functional.

Fixed by introducing a round-tracking parameter that dynamically updates the round counter on each click — allowing the button to progress through all rounds and display results after each.

// Reflection

Key Learnings

01

Building a real app requires architecture, not just code

Splitting the app into four separate modules (Pokédex, Quiz, Battle Sim, League) forced real thinking about structure — how modules talk to each other, how state is shared, and how the GUI connects everything.

02

Silent bugs are the hardest to find

The music bug produced no error — the app just ran silently. Learning to debug without error messages — tracing logic, checking initialisation order, and isolating components — was the most valuable debugging lesson of the module.

03

GUI state management is a discipline of its own

The Battle League "Next Round" bug revealed how easy it is to lose track of state in an event-driven GUI. Tracking round numbers explicitly with parameters — rather than implicitly — fixed it and taught a principle that applies far beyond Python.

04

A structured plan produces a better product

The 8-day timeline — with one module per development day — meant no feature was rushed or squeezed. Planning the build before writing any code was the single biggest difference between Code Lab 1 and Code Lab 2.

05

User experience matters even in desktop Python apps

Background music, consistent theming, and clear navigation turned a functional script into something that felt like a real product. Polish isn't decoration — it's the difference between a program and an application.
